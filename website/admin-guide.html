How Everything Works

End-to-end flow from a new lead calling to a job being booked in the calendar.

The Full Flow

Customer calls business number
→
Unanswered call hits Twilio
↓
Twilio fires /webhook/voice
→
Call rejected (silent), SMS triggered
↓
Claude AI generates first reply
→
SMS sent from client's Twilio number
↓
Customer replies
→
/webhook/sms → AI continues conversation
↓
Booking confirmed → Google Calendar event created
↓
Client gets SMS + email alert
→
Lead logged in portal
→
Zapier webhook fires (if set up)

Other Entry Points

Google Form / Contact Form via Zapier Pro

Zapier sends a POST to /webhook/zapier/:widgetKey — AI texts the customer back using the client's number and config. Same conversation flow as a missed call.

Website Chat Widget Pro

Visitor chats on the client's website → /api/widget/chat → AI responds in real time. No SMS — web only. Bookings land in calendar.

Client Lifecycle

What happens at each stage from signup to live bot.

  1. Client fills signup form + pays via Stripe Stripe fires checkout.session.completed → partial client record created in DB → setup email sent with a 7-day token link.
  2. Client fills setup form at /setup?token=… They enter trade, hours, area, pricing, FAQs, and create a portal password. Takes ~3 minutes.
  3. Auto-provisioning fires Twilio number purchased (780/825 area code) → bot activated → widget key generated (Pro) → "you're live" email sent with number + call forwarding instructions.
  4. Client sets up call forwarding They forward unanswered calls from their business phone to their Swiftbooked number. This is the only manual step for the client — see Call Forwarding Help.
  5. You connect their Google Form (Pro, if they have one) Client replies to the "you're live" email with their form link. You set up Zapier — see Zapier Setup. Takes ~10 minutes.
  6. Bot is running 24/7 Every missed call and form submission gets an instant AI response. You get weekly digest emails showing lead and booking counts.

When a Subscription Cancels

Stripe fires customer.subscription.deleted → client's active flag set to false → bot stops responding → you get an email notification. The Twilio number stays assigned — release it manually in the Twilio console if you want to reuse it.

Zapier Setup — Google Form → AI Text

When a Pro client replies with their Google Form link, follow these steps. Takes about 10 minutes. You'll need access to their Google account (or they can share the form with you).

First, get the client's widget key from the Admin Panel. Their webhook URL is: https://swiftbooked.ca/webhook/zapier/{widgetKey}
  1. Log in to zapier.com Use your Zapier account (one account can run Zaps for all clients).
  2. Click "Create Zap"
  3. Trigger: Google Forms → New Form Response Connect to the Google account that owns the form → select the client's specific form from the dropdown. Click "Test trigger" — it needs at least one previous submission to test with.
  4. Action: Webhooks by Zapier → POST In the URL field, paste the client's webhook URL. Set Content Type to application/json.
  5. Map the Data fields In the "Data" section, add these key-value pairs:
    name → [Form field: Name / Full Name] phone → [Form field: Phone / Phone Number] message → [Form field: Message / Notes] (optional) service → [Form field: Service Type] (optional)
    Only phone is required — the AI can still run without name/message.
  6. Test the action Zapier will POST to the webhook. You should see a 200 response with {"success":true}. If you see 404, the widget key is wrong.
  7. Publish the Zap Name it something clear like "Apex HVAC — Form → Swiftbooked".
Webhooks by Zapier requires a paid Zapier Starter plan (~$20/mo USD). Use Make (make.com) instead — it's free, supports webhooks, and works just as well. See the Make setup below.

Alternative: Make (make.com) — Free

Make is the recommended free option. 1,000 operations/month on the free plan — plenty for any client starting out. Make calls workflows "scenarios" instead of "Zaps" but the logic is identical.

  1. Go to make.com and sign up free Create an account — no credit card needed.
  2. Click "Create a new scenario"
  3. Add the first module: Google Forms → Watch Responses Search for "Google Forms" → select Watch Responses → connect the client's Google account → select their form. Set the limit to 1 (process one response at a time).
  4. Add the second module: HTTP → Make a Request Click the + after Google Forms → search "HTTP" → select Make a Request. Fill in:
    URL: https://swiftbooked.ca/webhook/zapier/{widgetKey} Method: POST Body type: Raw Content type: application/json Request content: {"name":"{{1.Name}}","phone":"{{1.Phone Number}}","message":"{{1.Message}}"}
    Replace the field names in {{ }} with the actual field names from the client's form — Make will show you available fields as you type.
  5. Click "Run once" to test Make will process the most recent form submission. You should see a green tick on both modules and a 200 response from the HTTP module.
  6. Turn the scenario on Toggle the switch in the bottom left to ON. Set the schedule to "Immediately" or every 15 minutes — immediately means it checks for new responses in real time.
Make's free plan runs scenarios every 15 minutes by default. To get near-instant responses, set the scheduling interval to the minimum (15 min is fine — client gets a text within 15 min of submitting, which is still much faster than a human checking email).

Website Contact Form — Email Notification Method

If a client has a website contact form (Squarespace, Wix, WordPress, etc.) and doesn't want to give you website access, the easiest path is the Email Parser by Zapier method. The client just forwards their form notification emails — you handle everything else.

What to ask the client: "What email address do your contact form notifications get sent to?" That's all you need from them.
  1. Go to parser.zapier.com and create a mailbox Zapier gives you a unique email address like abc123@robot.zapier.com. Copy it.
  2. Ask the client to forward form emails to that address They go into their email settings (Gmail, Outlook, etc.) and set up a forwarding rule: any email from their website form gets forwarded to the Zapier mailbox. Usually takes them 2 minutes — send them this:

    "In your email, set up a rule/filter to forward any emails from [their form notification address] to: [zapier mailbox address]. That's all you need to do."
  3. Train the Email Parser on a sample email Trigger a test form submission so an email lands in the Zapier mailbox. Open parser.zapier.com → click the email → highlight the customer's name, phone number, and message and label each one. Zapier learns the format and will extract those fields automatically from every future email.
  4. Create a Zap: Email Parser → Webhook POST Trigger: Email Parser by Zapier → New Email → select your mailbox.
    Action: Webhooks by Zapier → POST → URL: https://swiftbooked.ca/webhook/zapier/{widgetKey}
    Data:
    name → [Parsed field: Name] phone → [Parsed field: Phone] message → [Parsed field: Message]
  5. Test and publish Send a test form submission → confirm a 200 response → publish the Zap. Done.
Email Parser works on any Zapier paid plan. The free plan doesn't include it — use the $20/mo Starter plan which covers all clients in one account.

WordPress / Wix / Squarespace / Webflow

All major website builders have a native Zapier integration. The Zap is the same — just change the trigger from "Google Forms" to their platform:

  • WordPress (WPForms / Gravity Forms / Contact Form 7): WPForms and Gravity Forms have direct Zapier triggers. Contact Form 7 needs a plugin called "CF7 Zapier" (free, install from WordPress plugins).
  • Wix: Zapier → trigger: Wix → New Form Submission
  • Squarespace: Zapier → trigger: Squarespace → New Form Submission
  • Webflow: Zapier → trigger: Webflow → Form Submission

Everything after the trigger (the HTTP POST action and field mapping) stays exactly the same.

Custom-Built Website

If the client has a custom site, their web developer just needs to add one line to the form's submit handler. Send them this message:

// Send this to their developer: When the contact form submits, POST the form data to this URL: https://swiftbooked.ca/webhook/zapier/{widgetKey} Expected fields (JSON): name — customer's name phone — customer's phone number (required) message — their message or notes (optional) service — service type if you have that field (optional) Example fetch call: fetch("https://swiftbooked.ca/webhook/zapier/{widgetKey}", { method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify({ name, phone, message, service }) });

Takes a developer about 5 minutes. The client texts you their developer's contact and you send them the above.

What the AI receives

The inbound webhook builds an opening message like: "I need help with: HVAC repair. My furnace isn't working. My name is Mike" — then runs it through the client's AI config. The customer gets a text from the client's Twilio number within seconds.

Zapier Setup — Booking Confirmed → CRM / Google Sheets

When the AI confirms a booking, Swiftbooked can automatically push the details to a Google Sheet, HubSpot, QuickBooks, or any app Zapier supports.

  1. In Zapier, create a new Zap Trigger: Webhooks by Zapier → Catch Hook. Copy the unique webhook URL Zapier gives you — it looks like https://hooks.zapier.com/hooks/catch/…
  2. Save that URL to the client's record Use this API call (or add a field in the admin panel):
    PATCH /api/portal/integrations Authorization: Bearer {client JWT} Body: { "zapier_webhook_url": "https://hooks.zapier.com/..." }
    Or just update it directly in the database via the admin panel's edit modal once you add the field.
  3. Set up the Zapier action The payload Swiftbooked sends on each booking:
    { "event": "booking_confirmed", "business": "Apex Heating & Cooling", "customer_phone": "+17805551234", "booking_id": "BK7X2R", "timestamp": "2026-04-24T14:32:00.000Z" }
    Map these to whatever app you're connecting — Google Sheets row, HubSpot contact, QuickBooks invoice, etc.
  4. Test with a real booking The webhook only fires when the AI confirms a booking (result.booked = true), not on every message.

Call Forwarding Help

This is the one thing clients need to do themselves. If a client says their bot isn't responding to calls, this is almost always the issue.

iPhone (any carrier)

Settings → Phone → Call Forwarding → toggle on → enter the Swiftbooked number. Only "When Busy" or "No Answer" forwarding is needed — not unconditional.

Bell / Telus (any phone)

Dial and press call: *61*+1{swiftbooked_number_without_plus}#

To disable voicemail first (so bot answers before voicemail): dial #404# and press call.

Rogers (any phone)

Dial and press call: *61*+1{swiftbooked_number_without_plus}#

To disable voicemail: dial #BAL# and press call.

Business VoIP / Ring Central / 8x8

Each system is different. Have the client log into their VoIP dashboard → Call Forwarding → "When unanswered" → enter the Swiftbooked number. If they're stuck, ask them which VoIP provider they use and look up the specific steps.

Testing It

Call the client's business number from a different phone. Let it ring until it would normally go to voicemail. Within 30 seconds, the phone you called from should receive a text from the Swiftbooked number. If it doesn't, call forwarding isn't set up correctly.

Manual Twilio Number Assignment

Auto-provisioning works almost every time. If it fails, here's what to do.

You'll get an email notification if auto-provisioning fails, with an "Open Admin Panel" link. The client's account is set up but inactive until a number is assigned.
  1. Log in to console.twilio.com Go to Phone Numbers → Manage → Buy a number → Canada → search for 780 or 825 area code → buy one.
  2. Configure the number Set Voice webhook to: https://swiftbooked.ca/webhook/voice (POST)
    Set SMS webhook to: https://swiftbooked.ca/webhook/sms (POST)
  3. Assign it in the admin panel Find the client → Edit → enter the Twilio number in E.164 format: +17805551234 → set Active to true → save.
  4. Email the client their number Send them the number and call forwarding instructions manually (copy from Call Forwarding Help).

Website Chat Widget Pro Only

The widget is auto-generated on setup completion for Pro clients. Their "you're live" email already contains the code snippet. Here's what to do if they need help installing it.

The snippet

<script src="https://swiftbooked.ca/widget.js" data-key="sb_xxxxxxxx"></script>

Goes just before </body> on any page they want the chat to appear. Works on WordPress, Wix, Squarespace, Webflow, or any custom site.

WordPress (most common)

  1. In WordPress admin: Appearance → Theme Editor (or use a plugin like "Insert Headers and Footers")
  2. Paste the script tag in the footer section
  3. Save. The chat bubble should appear on all pages immediately.

Finding a client's widget key

Go to the Admin Panel → find the client → their widget_key is shown in the table. To regenerate a key, use the API: POST /api/admin/clients/{id}/widget-key (requires admin auth). Note: regenerating breaks any existing installation until they update the snippet.

The widget talks to

GET /api/widget/config?key=… — fetches business name for greeting
POST /api/widget/chat — handles chat messages (same rate limiting as demo chat)

Calendar Setup

Google Calendar (most common)

Clients connect their calendar during the setup form. The setup page has a "Connect Google Calendar" button that runs the OAuth flow. Once connected, a refresh token is stored in the DB — the AI uses it to check availability and create events.

To check if a client has it connected: Admin Panel → look for the green calendar badge. No badge = no token = bookings won't go to their calendar (AI will still respond, just won't auto-book).

To reconnect a calendar: send them to /connect-calendar/{clientId} — they'll go through Google OAuth again.

Calendly

Client provides their Calendly URL during setup (e.g. calendly.com/theirbusiness/service). The AI shares this link in the conversation instead of booking directly. No OAuth needed — just the URL.

Outlook / other

Not directly integrated yet. Workaround: set up a Calendly account connected to their Outlook and use the Calendly URL method.

Troubleshooting

Bot isn't texting back after missed calls
  1. Check the client's call forwarding is set up correctly
  2. Check their active flag is true and they have a twilio_number assigned (admin panel)
  3. Call the Swiftbooked number directly — if it texts back, call forwarding is the issue
  4. Check Railway logs for errors on /webhook/voice
Emails not sending
  1. Check RESEND_FROM_EMAIL env var is set correctly
  2. In Resend dashboard → Domains — confirm swiftbooked.ca is verified (green)
  3. Check Railway logs for [Email error]
  4. Temporary fallback: change RESEND_FROM_EMAIL to Swiftbooked <onboarding@resend.dev> — only delivers to your verified Resend email address
Twilio auto-provision failed

You'll get an email. Follow Manual Twilio Number Assignment steps above.

Zapier inbound webhook returning 404

The widget key in the URL is wrong. Double-check the client's widget_key in the admin panel. Remember — only Pro clients have widget keys.

Client can't log into portal
  1. Check setup_completed is true — if false, setup form wasn't submitted
  2. Check password_hash is not null
  3. Use the admin panel to reset their password: POST /api/admin/clients/{id}/password
Booking confirmed but not showing in Google Calendar
  1. Check google_refresh_token is not null (admin panel)
  2. If null, send client to /connect-calendar/{clientId} to reconnect
  3. Check Railway logs for [Calendar] errors
  4. If they use Calendly, make sure calendly_url is filled in — that takes priority over Google Calendar direct booking
Checking Railway logs

Railway Dashboard → your project → Deployments → click the active deployment → Logs. Filter by keywords like [Email error], [Twilio], [Zapier], [Calendar].
